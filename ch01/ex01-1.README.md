## 1. 알고리즘 설계와 코딩 중 무엇이 더 중요한가?

**알고리즘 설계**가 더 중요하다.

알고리즘은 문제를 해결하는 절차와 방법으로,  
설계가 잘못되면 코딩을 아무리 잘해도 올바른 결과를 얻을 수 없기 때문이다.  
코딩은 설계된 알고리즘을 C++ 언어로 표현하는 수단에 불과하다.

## 2. 클래스와 객체란 무엇인가?

- **클래스(Class)** : 객체를 만드는 틀로, 데이터(멤버 변수)와 함수(멤버 함수)를  
  하나로 묶은 사용자 정의 자료형이다.

- **객체(Object)** : 클래스라는 틀에서 생성된 실체(instance)이며 클래스형 변수이다.

```cpp
class Circle {          // 클래스 (틀)
private:
    int radius;
public:
    double getArea() { return 3.14 * radius * radius; }
};

Circle c1;              // 객체 (실체)
```

## 3. 절차지향 vs 객체지향 프로그래밍 언어의 차이

| 구분 | 절차지향 | 객체지향 |
|------|----------|----------|
| 방식 | 실행할 절차대로 명령어를 나열 | 객체를 정의하고 객체들의 상호작용으로 구현 |
| 설계 | 흐름도(Flowchart) 중심 | 객체 관계도 중심 |
| 특징 | 수학 계산, 통계 처리에 적합 | 실세계 모델링, 대규모 소프트웨어에 적합 |
| 대표 언어 | C, Pascal, Fortran | C++, Java, Python |

## 4. 라이브러리(Library)란?

프로그램 개발 시 자주 사용되는 기능(printf, cin, cout 등)을  
미리 작성해 놓은 코드의 집합이다.

C++ 표준 라이브러리는 다음 세 가지로 구성된다.

| 종류 | 설명 | 예시 |
|------|------|------|
| C 라이브러리 | 기존 C 표준 라이브러리를 C++에서 사용 가능하게 한 함수들 | `cstdio`, `cstring` |
| C++ 입출력 라이브러리 | 콘솔 및 파일 입출력을 위한 라이브러리 | `iostream`, `fstream` |
| C++ STL 라이브러리 | 제네릭 프로그래밍을 지원하는 템플릿 라이브러리 | `vector`, `map`, `list` |

---

## 5. 라이브러리가 목적파일 형태로 존재하는 이유

라이브러리는 이미 컴파일된 **목적파일(.lib / .a)** 형태로 존재한다.

- **소스코드 보호** : 내부 구현 코드를 공개하지 않고 기능만 제공할 수 있다.
- **컴파일 시간 단축** : 매번 다시 컴파일할 필요 없이 링크 과정에서 바로 연결하여 사용할 수 있다.
- **재사용성** : 한 번 컴파일된 목적파일을 여러 프로젝트에서 반복적으로 사용할 수 있다.

---

## 6. 링크 과정이 필요한 이유

컴파일 단계에서 생성된 목적파일(`.obj`)은 바로 실행할 수 없다.

링크 과정은 내가 작성한 목적파일과 C++ 표준 라이브러리 목적파일을  
**링커(Linker)** 가 하나로 연결하여 최종 실행파일(`.exe`)을 만드는 과정이기 때문이다.

```
hello.obj  +  C++ 표준 라이브러리(cout, << 등)
                      ↓ 링커(Linker)
                  hello.exe
```

# C++ 생성자(Constructor) 과제

---

## 1. 생성자를 통해 멤버변수를 초기화하는 이유

객체가 생성될 때 멤버변수에는 이전 메모리에 남아있던 쓰레기값이 들어있다. 이 상태로 객체를 사용하면 잘못된 연산 결과나 예측 불가능한 버그가 발생할 수 있다.

생성자는 객체가 만들어지는 순간 자동으로 호출되는 특별한 함수이기 때문에, 생성자 안에서 멤버변수를 초기화하면 객체가 생성됨과 동시에 항상 안전한 초기 상태가 보장된다. 개발자가 따로 초기화 코드를 작성하거나 호출하는 것을 잊어버릴 가능성이 없어지므로 안전성과 일관성이 높아진다.

또한 `const`나 참조(`&`) 멤버변수는 선언 이후 값을 변경할 수 없기 때문에, 반드시 생성자에서만 초기화가 가능하다는 점도 생성자를 통한 초기화가 필요한 이유 중 하나이다.

---

## 2. 생성자 몸체보다 멤버 초기화 리스트를 사용하는 이유

생성자 몸체에서 멤버변수에 값을 넣는 것은 엄밀히 말하면 **초기화가 아니라 대입**이다. 멤버변수는 생성자 몸체가 실행되기 전에 이미 생성되므로, 몸체에서 값을 넣는 것은 이미 생성된 변수에 값을 덮어쓰는 2단계 과정이 된다.

반면 멤버 초기화 리스트를 사용하면 멤버변수가 생성되는 순간 바로 원하는 값으로 초기화되므로 1단계로 처리된다. 이는 불필요한 과정을 줄여 성능상 더 효율적이다.

특히 `const` 멤버변수나 참조(`&`) 멤버변수는 생성 이후 값을 바꿀 수 없으므로, 생성자 몸체에서 대입하면 컴파일 에러가 발생한다. 이러한 멤버들은 반드시 멤버 초기화 리스트에서만 초기화할 수 있다.

따라서 특별한 이유가 없는 한 항상 멤버 초기화 리스트를 사용하는 것이 권장된다.

---

## 3. 위임 생성자와 타겟 생성자로 나누어 작성하는 이유

하나의 클래스에 여러 생성자를 만들다 보면, 각 생성자마다 비슷한 초기화 코드가 반복되는 문제가 생긴다. 코드가 중복되면 나중에 초기화 로직을 수정할 때 모든 생성자를 일일이 찾아서 고쳐야 하므로 실수가 발생하기 쉽고 유지보수가 어려워진다.

이 문제를 해결하기 위해 **타겟 생성자**에 실제 초기화 작업을 모두 집중시키고, 나머지 생성자들은 타겟 생성자에게 초기화를 맡기는 **위임 생성자**로 작성한다. 이렇게 하면 초기화 로직이 한 곳에만 존재하게 되어 코드 중복이 제거되고, 수정이 필요할 때 타겟 생성자 하나만 고치면 된다.

결과적으로 코드의 중복 제거, 유지보수 편의성 향상, 버그 방지, 그리고 각 생성자의 역할이 명확히 분리되는 가독성 향상이라는 이점을 얻을 수 있다.

---

## 4. 함수 몸체에서 생성자를 직접 호출하면?

생성자는 객체가 처음 만들어질 때 딱 한 번 자동으로 호출되도록 설계된 특별한 함수이다. 따라서 이미 생성된 객체에서 생성자를 직접 이름으로 호출하려 하면 컴파일 에러가 발생한다.

만약 `객체 = 클래스이름(인자)` 형태로 호출하면 컴파일은 되지만, 이는 생성자를 직접 호출하는 것이 아니라 **새로운 임시 객체를 생성한 후 기존 객체에 대입**하는 동작이다. 즉 원래 객체를 다시 초기화하는 것이 아니므로 의도한 동작과 다를 수 있다.

생성자를 다시 호출하고 싶은 상황이 생긴다면, 생성자를 직접 호출하는 대신 별도의 초기화 멤버함수(예: `reset()`, `init()`)를 만들어 사용하는 것이 올바른 방법이다.

| 주제 | 핵심 내용 |
|---|---|
| 생성자로 초기화하는 이유 | 쓰레기값 방지, 객체 생성 시 자동 호출로 안전성과 일관성 보장 |
| 멤버 초기화 리스트 권장 이유 | 대입(2단계)이 아닌 초기화(1단계), const/참조 멤버 초기화 가능 |

| 몸체에서 생성자 직접 호출 | 컴파일 에러 또는 임시 객체 생성 후 대입 — reset 함수로 대체 권장 |

## Q1. 함수를 사용하는 이유를 설명하라

함수란 **자주 쓰는 기능을 모아놓은 코드 단위**로, 다음과 같은 이점 때문에 사용한다.

- **코드 중복 방지** : 동일한 로직을 여러 곳에 반복해서 작성하지 않아도 되므로 전체 소스코드의 크기가 줄어든다.
- **재활용성 향상** : 한 번 작성한 함수를 프로그램의 여러 곳에서, 또는 다른 프로젝트에서도 재사용할 수 있어 개발 기간을 단축할 수 있다.
- **가독성 및 유지보수 향상** : 기능별로 함수로 분리해두면 코드를 이해하기 쉽고, 수정이 필요할 때 해당 함수만 고치면 된다.

## Q2. 짧은 코드를 갖는 함수를 자주 호출하면 발생하는 오버헤드를 설명하라

함수를 호출할 때 컴파일러는 실제 함수 몸체 코드(T2) 외에도 **자동으로 추가되는 처리 코드**를 생성한다.

| 단계 | 설명 |
|------|------|
| **T1** (호출 전) | 현재 CPU 레지스터 값 저장, 리턴 주소 저장, 매개변수 할당 및 초기화 |
| **T2** (실행) | 함수 몸체의 실제 코드 실행 |
| **T3** (반환 후) | 저장했던 레지스터 값 복원, 리턴 값을 임시 저장소에 저장, 리턴 주소로 복귀 |

**함수 실행에 걸리는 총 시간 = T1 + T2 + T3**이며, T1+T3이 **오버헤드(overhead)** 에 해당한다.

함수 몸체가 짧을수록 T2는 매우 작아진다. 그런데 T1+T3은 함수 크기와 무관하게 **항상 발생**하므로, 짧은 함수를 자주 호출할수록 실질적인 실행 시간 대비 오버헤드의 비중이 커진다.

> **예시** : `odd(i)` 함수(1줄 코드)를 10,000번 호출할 때, T1+T3 = 1ms라고 가정하면  
> 오버헤드 = 10,000 × 1ms = **10초** → 함수를 쓰는 이점보다 손해가 훨씬 크다.

특히 C++에서는 생성자, 소멸자, getter/setter 같이 **코드가 짧고 호출 빈도가 높은 멤버 함수**가 많기 때문에 이 문제가 더욱 심각해진다.

## Q3. 인라인 함수 치환 시 private 멤버 변수를 직접 접근하게 되는데, 문법 오류가 아닌가?

**문법 오류가 아니다.**

인라인 함수의 치환은 **소스코드 수준의 텍스트 치환이 아니라, 컴파일러가 기계어(목적 코드) 수준에서 수행하는 치환**이다.

즉, 소스코드 상에서는 여전히 `cir.getArea()`라는 **멤버 함수 호출 형태**가 유지된다. 컴파일러는 이 코드의 **접근 권한(private/public)을 소스코드 단계에서 검사**하고, 문법적으로 이상이 없음을 확인한 뒤에, 기계어로 변환하는 과정에서 함수 호출 대신 해당 코드를 직접 삽입한다.

따라서 private 접근 규칙은 **소스코드 컴파일 단계에서 이미 검증 완료**되며, 기계어 수준의 치환은 그 이후의 일이므로 접근 지정자 위반이 발생하지 않는다.

## Q4. 인라인 함수는 처음부터 함수를 안 쓰면 안 되나요?

**안 된다.** 그 이유는 다음과 같다.

### 함수를 쓰지 않으면 발생하는 문제

- **코드 중복** : 같은 로직을 여러 곳에 반복 작성해야 해서 소스코드가 길어지고 유지보수가 어려워진다.
- **캡슐화·정보은닉 불가** : OOP의 핵심 개념인 캡슐화와 정보은닉은 멤버 함수를 통해 private 데이터를 간접 접근하는 구조로 이루어진다. 함수 없이 직접 접근하면 이 원칙이 무너진다.
- **가독성·재활용성 저하** : 로직이 흩어져 있어 코드를 이해하기 어렵고, 재사용이 불가능하다.

### 인라인 함수가 존재하는 이유

| 구분 | 소스코드 | 기계어(실행 파일) |
|------|----------|-------------------|
| **일반 함수** | 함수 존재 | 함수 존재 (호출 오버헤드 발생) |
| **인라인 함수** | 함수 존재 | 함수 없음 (코드 직접 삽입) |
| **함수 미사용** | 함수 없음 | 함수 없음 |

인라인 함수는 **"소스코드에서는 함수로 작성하여 OOP의 장점(캡슐화, 정보은닉, 재사용성)을 유지하면서도, 실행 파일에서는 함수를 없애 성능을 높이는"** 절충안이다.

결론적으로 인라인 함수는 **개발 편의성(함수의 장점)**과 **실행 성능(함수 미사용의 장점)**을 동시에 얻기 위해 설계된 기법이다.
